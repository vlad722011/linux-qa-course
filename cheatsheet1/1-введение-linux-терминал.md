 Глава 1. Введение в Linux и
терминал
1.1Почему Linux
•Linux — это свободная, многопользовательская, многозадачная
операционная система с открытым исходным кодом.
•Она используется на серверных инфраструктурах, VPS, CI/CD, в контейнерах,
встраиваемых системах.
•Принципиальное отличие: всё в Linux — это файл или процесс, и система
прозрачна для изучения.
•Для QA-инженера Linux ценен тем, что позволяет:
1.Диагностировать поведение приложений и тестов на уровне системы.
2.Контролировать процессы, ресурсы и права.
3.Работать с серверами без GUI, что типично для CI/CD и удалённых сред.
Ключевая мысль: Linux — это инструмент, а shell — язык общения с этим инструментом.
1.2Shell vs GUI
•GUI (графический интерфейс) — визуальный способ управления, удобен на
десктопах.
•Shell — это командная строка, основной инструмент управления системой
на сервере.
•Shell не только запускает команды — он соединяет процессы и управляет
потоками данных.
•Даже когда кажется, что “команда просто выводит что-то на экран”, на самом
деле shell:
•Создаёт процесс
•Связывает stdout/ stderr / stdin
•Управляет окружением
## Пример:
cat /var/log/syslog | grep ssh | tail -n 20
•cat создаёт процесс, читает файл и отдаёт данные в stdout
•grep ssh создаёт процесс, получает поток и фильтрует строки

•tail -n 20 выводит последние 20 строк

Это цепочка процессов и потоков, а не “магия команды”.
1.3Терминалы и сессии
•Терминал — это интерфейс для общения с shell.
•Каждое подключение SSH создаёт новую сессию: свои переменные окружения,
текущий каталог и процессы.
•tty — терминальное устройство, с которым работает shell.
•Даже закрыв терминал, процессы могут продолжать работать в фоне.
•Терминалы можно переключать с помощью Ctrl+Alt+F1–F6 (локально)
или открывать несколько SSH-сессий.
Команды для работы с сессиями:
who# кто подключён
w# кто подключён и что делает
tty# показывает устройство текущей сессии
1.4Потоки данных: stdin, stdout, stderr
•stdin — стандартный вход (обычно клавиатура или поток от другой команды).
•stdout — стандартный вывод (обычно экран или следующая команда).
•stderr — поток ошибок.
Примеры перенаправления:
echo "hello" > output.txt     # записать stdout в файл
echo "error" 2> error.log     # записать stderr в файл
echo "both" &> both.log         # stdout+stderr вместе
cat file | grep pattern > result.txt   # stdout одного процесса в stdin другого
•Понимание потоков критично для QA: можно фильтровать логи, направлять ошибки
в отдельные файлы, анализировать поведение приложений.

1.6Особенности работы с терминалом
•История команд:
history# список последних команд
!n# выполнить команду Non из истории
!!# повтор последней команды
•Alias:
alias ll='ls -lah'
•Автодополнение: Tab — ускоряет работу и снижает ошибки.
•Разные shell:
•bash — стандарт, чаще всего на серверах
•zsh — более функциональный, с автодополнением и подсветкой
•sh — минимальный shell, используется в скриптах
•Окружение: каждая сессия имеет свои переменные: $HOME, $USER, $PATH.
•Переменные можно посмотреть:
env
printenv
1.7Практический кейс для QA
Задача: подключение к серверу и диагностика терминала.
1.Подключаемся по SSH:
ssh user@vps_ip
2.Проверяем текущий каталог:

pwd
3.Просматриваем файлы, включая скрытые:
ls -lah
4.Проверяем переменные окружения:
env | grep HOME
echo $PATH
5.Пробуем цепочку потоков:
cat /var/log/syslog | grep ssh | tail -n 20
•Видим, что вывод логов можно фильтровать без GUI.
•QA-практика: проверка, где происходят ошибки, и понимание, что процессы
shell создают поток данных.
 Итог первой главы:
•Linux — инструмент, shell — язык управления.
•Терминал создаёт сессии с собственными процессами, потоками и окружением.
•stdout, stderr, stdin — потоки, которые можно перенаправлять и комбинировать.
•Первые команды (ls, pwd, cd, echo) — это базовые кирпичики, из которых
строится понимание системы.
•История команд, alias, автодополнение — повышают эффективность работы.
•Для QA это критично: мы можем отслеживать поведение системы и процессов
на уровне данных и потоков.
