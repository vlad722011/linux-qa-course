# Guide 05a: PATH и профили bash — разбор реальной сессии (QA)

Эта глава — “разбор полётов” после практики: как реально формируется PATH, почему команда может запускаться “не та”, и почему по SSH работает, а в CI — нет.

---

## 1) PATH простыми словами
`PATH` — переменная окружения со списком каталогов, разделённых двоеточием `:`.

Когда ты вводишь команду без полного пути (например `docker`), shell:
1) идёт по каталогам из PATH **слева направо**,
2) ищет исполняемый файл с таким именем,
3) запускает **первое** найденное совпадение.

Отсюда правило: порядок PATH — это политика приоритетов.

---

## 2) Как увидеть, что реально запускается
### 2.1 Команда “покажи все совпадения”
~~~bash
type -a docker
type -a bash
type -a ssh
~~~

`type -a` показывает все найденные варианты команды в каталогах PATH (и порядок поиска).

### 2.2 Почему иногда видно /usr/bin и /bin одновременно
Если в PATH есть и `/usr/bin`, и `/bin`, то `type -a` покажет оба пути.

Важно: это не обязательно “две разные версии”.
Часто один путь является ссылкой на другой или оба указывают на один и тот же файл (зависит от системы).

---

## 3) Демо: две версии одной команды
Цель: увидеть, как порядок PATH меняет “победителя”.

Создай две команды с одним именем `hello`:
~~~bash
mkdir -p ~/bin ~/altbin

cat > ~/bin/hello <<'EOT'
#!/usr/bin/env bash
echo "HELLO from ~/bin"
EOT
chmod +x ~/bin/hello

cat > ~/altbin/hello <<'EOT'
#!/usr/bin/env bash
echo "HELLO from ~/altbin"
EOT
chmod +x ~/altbin/hello
~~~

Теперь переключай порядок PATH:
~~~bash
export PATH="$HOME/bin:$HOME/altbin:$PATH"
type -a hello
hello

export PATH="$HOME/altbin:$HOME/bin:$PATH"
type -a hello
hello
~~~

Вывод должен меняться — это и есть приоритет PATH.

---

## 4) Важное: `export PATH=...` живёт только в текущей сессии
Если ты сделал `export PATH=...` в терминале, это изменение:
- действует в текущем shell и его дочерних процессах,
- пропадает после закрытия SSH/терминала.

Файлы `~/bin/hello` останутся на диске, но команда по имени будет находиться только если `~/bin` есть в PATH.

---

## 5) Почему в новой SSH-сессии PATH “сам” изменился
На Ubuntu часто в `~/.profile` есть правило:

- “если существует каталог `$HOME/bin`, добавить его в PATH”.

Проверка:
~~~bash
grep -nH -E '(\$HOME|~)/bin|HOME/bin' ~/.profile ~/.bashrc ~/.bash_profile 2>/dev/null || true
~~~

Типичная логика в `~/.profile` (по смыслу):
~~~bash
if [ -d "$HOME/bin" ] ; then
    PATH="$HOME/bin:$PATH"
fi
~~~

Поэтому: создал `~/bin` → в следующих login-сессиях `~/bin` появляется в PATH автоматически.

---

## 6) Ошибка новичка: “запустил ~/.profile”
Команда:
~~~bash
~/.profile
~~~
пытается запустить файл как программу.

Обычно `~/.profile` не исполняемый (нет `x`), поэтому будет `Permission denied`.
Правильно:
- открыть: `nano ~/.profile` (без sudo, если файл твой),
- применить в текущей сессии: `source ~/.profile` (или `. ~/.profile`).

Проверить права:
~~~bash
ls -l ~/.profile
~~~

---

## 7) Login PATH vs “clean PATH” (QA-инсайт для CI)
Одна из главных причин “по SSH работает, а в CI нет” — разные startup files.

Сравнить:
~~~bash
echo "$PATH"

env -i bash --noprofile --norc -c 'echo "$PATH"'
~~~

- В login-сессии PATH обычно богаче (включает `~/bin`, `snap/bin`, и т.п.).
- В clean bash PATH обычно минимальный.

---

## 8) Мини-чеклист (как не словить боль)
- Перед важной работой: `type -a tool` и `tool --version`.
- Не переписывай PATH целиком без необходимости: добавляй через `PATH="NEW:$PATH"`.
- Если нужно избежать конфликтов версий: разные имена команд, `/opt/tool/current/bin`, либо контейнеры/окружения проекта.
