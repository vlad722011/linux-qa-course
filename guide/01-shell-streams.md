# 01 — Shell + streams (stdin/stdout/stderr)

## Зачем это QA-инженеру
Когда тесты падают, ты почти всегда работаешь с выводом команд: логи, ошибки, отчёты, диагностика окружения.
Навык “управлять выводом” превращает хаос в артефакты: отдельный файл с результатом, отдельный файл с ошибками, один общий лог для CI.

Практически: если ты умеешь stdout/stderr и редиректы, ты умеешь:
- собрать “доказательства” в файл и приложить к багу/тикету;
- сделать воспроизводимый запуск скрипта в CI;
- быстро отфильтровать шум и оставить только нужные строки.

Самопроверка:
1) Чем stdout отличается от stderr по смыслу?
2) Почему ошибки не стоит смешивать с “нормальным” выводом, когда ты делаешь отчёт?

## Модель в голове: 3 потока
У процесса (команды) есть три стандартных потока:
- stdin — откуда читаем (по умолчанию клавиатура/терминал);
- stdout — куда пишем нормальный результат (по умолчанию терминал);
- stderr — куда пишем ошибки/диагностику (по умолчанию терминал).

Важно: stdout и stderr — два независимых потока. Это значит, что их относительный порядок может “плавать” в общем логе, если ты их склеил.

Самопроверка:
1) Какие номера у потоков stdin/stdout/stderr?
2) Почему у stdout и stderr могут быть разные правила буферизации?

## Редиректы: как направлять вывод
Основные операторы оболочки:

1) `>` — отправить stdout в файл (и перезаписать файл).
Пример:
- `echo "hello stdout" > out.txt`

2) `>>` — отправить stdout в файл (и дописать в конец).
Это нужно, когда ты копишь лог нескольких шагов.
Пример:
- `echo "step 1" >> run.log`
- `echo "step 2" >> run.log`

3) `2>` — отправить stderr в файл.
Пример:
- `ls no_such_file 2> err.txt`

4) Склейка stdout + stderr в один файл (правильный вариант):
- `cmd > all.txt 2>&1`

Почему порядок важен:
- `> all.txt` сначала направляет stdout в файл,
- `2>&1` затем направляет stderr туда же, куда уже направлен stdout (то есть в all.txt).

Ловушка:
- `cmd 2>&1 > all.txt`
Здесь stderr сначала “прилипает” к текущему stdout (к терминалу), а только потом stdout уходит в файл.
Итог: stdout в файле, stderr в терминале.

Самопроверка:
1) Чем отличается `>` от `>>`?
2) Что окажется в файле при `cmd 2>&1 > all.txt`?

## Лабораторная (делать на VPS/локально)
Цель: руками увидеть stdout, stderr и их перенаправление.

1) Подготовка папки:
- `mkdir -p ~/lab1_streams && cd ~/lab1_streams`

2) stdout:
- `echo "hello stdout"`
- `echo "hello stdout" > out.txt`
- `cat out.txt`

3) stderr:
- `ls no_such_file`
- `ls no_such_file 2> err.txt`
- `cat err.txt`

4) Объединение потоков:
- `ls out.txt no_such_file > all.txt 2>&1`
- `cat all.txt`

Критерии зачёта:
- out.txt содержит только строку `hello stdout`.
- err.txt содержит сообщение об ошибке `ls` про несуществующий файл.
- all.txt содержит и успешный вывод, и ошибку.

## Типовые грабли (очень жизненно)
- “Почему ничего не вывелось?” — потому что stderr перенаправили в файл.
- “Почему порядок строк странный?” — потому что stdout/stderr независимы; после склейки порядок может зависеть от буферизации/момента записи.
- “Почему команды не запускаются без sudo?” — потому что установка пакетов/управление сервисами требует прав администратора (но работать лучше под обычным пользователем).
