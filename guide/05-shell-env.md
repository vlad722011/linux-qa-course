# Guide 05: Shell и окружение (переменные, PATH, export, rc-файлы) — QA

## Цель главы
После этой главы ты должен уметь:
- Понимать разницу: локальная переменная vs переменная окружения (export).
- Объяснять, почему команда/скрипт “работает в одном терминале”, но не работает в другом/в CI.
- Понимать, где настраивается окружение: `.bashrc`, `.profile` и почему их нельзя путать.
- Делать диагностические “снимки окружения” как артефакты.

---

## 1) Контекст: почему QA это критично
Большая часть “странных” багов в автотестах и CI — это не баги кода, а разные окружения:
- PATH отличается (команда не находится).
- Нет переменных (токенов/ENV/конфигов).
- Скрипт запускается не тем shell (`sh` вместо `bash`).
- Интерактивная сессия и non-interactive окружение ведут себя по-разному.

---

## 2) Переменные: локальные и окружения
### 2.1 Локальная переменная (только текущий shell)
~~~bash
MYVAR="hello"
echo "parent: $MYVAR"
bash -lc 'echo "child: $MYVAR"'
~~~

### 2.2 Переменная окружения (export) — видна дочерним процессам
~~~bash
export MYVAR="hello"
bash -lc 'echo "child: $MYVAR"'
~~~

Смысл: `export` помечает переменную для передачи дочерним процессам (то есть программам, которые ты запускаешь из shell). [web:142]

## 3) PATH: почему “command not found”
PATH — список каталогов, где shell ищет исполняемые файлы.

Посмотреть PATH:
~~~bash
echo "$PATH"
~~~

Понять, откуда берётся команда:
~~~bash
which bash
type -a bash
~~~

Важно: `which` — утилита, а `type` — встроенная команда shell, часто даёт более точную информацию.

## 4) Диагностика окружения (QA-паттерн)
Снять “снимок окружения” в файл (удобно прикладывать к тикету):
~~~bash
mkdir -p ~/qa-env && cd ~/qa-env
env | sort > env_snapshot.txt
printf "SHELL=%s\n" "$SHELL" > shell_info.txt
printf "0=%s\n" "$0" >> shell_info.txt
echo "$PATH" > path.txt
ls -lah
~~~

## 5) Startup files: .bashrc vs .profile (важно для CI)
У bash есть разные режимы запуска, и от этого зависит, какие файлы он читает.

### 5.1 Что обычно где
- `~/.bashrc` — настройки для интерактивного bash (когда ты открываешь терминал/ssh и работаешь руками).
- `~/.profile` или `~/.bash_profile` — настройки при логине (login shell).

Идея простая: “то, что нужно для интерактива” — в `.bashrc`, “то, что нужно при входе” — в `.profile`.
Подробнее о различиях часто объясняют так: `.bashrc` для интерактивных shell, `.profile`/`.bash_profile` для login shell. [web:137][web:140]

### 5.2 QA-правило
Не делай в `.bashrc` ничего, что может ломать non-interactive окружение (например, команды, которые требуют TTY, или шумный вывод).

## 6) Практика на VPS: увидь разницу своими глазами
### 6.1 Проверка: что видно в “дочернем bash”
~~~bash
MYX="noexport"
export MYE="exported"
bash -lc 'echo "MYX=$MYX MYE=$MYE"'
~~~
Ожидаемо: MYX пустой, MYE есть.

### 6.2 Проверка: env vs локальные переменные
~~~bash
MYLOCAL="x"
export MYENV="y"
( set | grep -E '^MY(L|E)'; echo "---"; env | grep -E '^MY(L|E)' )
~~~

## 7) QA-кейс: “локально работает, в CI — нет”
Симптомы:
- `command not found`
- “переменная пустая”
- “не тот python/java/node”
- “в отчёте нет логов”

Твой план диагностики:
1) Снять окружение:
~~~bash
env | sort > env_ci_like_snapshot.txt
~~~
2) Снять PATH:
~~~bash
printf "%s\n" "$PATH" > path_ci_like.txt
~~~
3) Проверить, какой shell:
~~~bash
echo "$SHELL" > shell_ci_like.txt
echo "$0" >> shell_ci_like.txt
~~~
4) Приложить эти 3 файла к тикету/пайплайну (как артефакты).
