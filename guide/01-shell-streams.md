# Guide 01: Shell + потоки (stdin/stdout/stderr) — база Linux для QA

## Цель главы
После этой главы ты должен уметь:
- Отличать терминал от shell и понимать, что shell делает “под капотом”.
- Понимать stdin/stdout/stderr и применять редиректы.
- Собрать простые диагностические отчёты (файлы-артефакты) для баг-репорта и CI.

---

## 1) Терминал, shell и сессия
Терминал — это интерфейс (окно/ssh-клиент), через который ты работаешь с shell.

Shell (обычно bash на серверах) — программа, которая:
- читает твою команду,
- запускает процессы,
- настраивает потоки ввода/вывода,
- управляет окружением,
- возвращает код завершения (0 — успех, не 0 — ошибка).

Каждое SSH-подключение создаёт отдельную сессию: свой текущий каталог, переменные окружения, историю команд и процессы.

### Практика (на VPS)
~~~bash
whoami
id
pwd
tty
echo "$SHELL"
echo "$0"
ps
~~~

---

## 2) Потоки: stdin/stdout/stderr
У каждого процесса есть 3 стандартных потока:
- stdin (0) — вход (клавиатура или поток из другой команды)
- stdout (1) — обычный вывод
- stderr (2) — вывод ошибок

QA-почему важно:
- В CI логи часто “размазаны”, и stderr помогает быстро найти причину падения.
- Можно сохранять stdout отдельно от ошибок, чтобы проще анализировать.

---

## 3) Редиректы (перенаправление вывода)
### 3.1 stdout в файл
~~~bash
echo "hello" > out.txt
echo "world" >> out.txt
cat out.txt
~~~

### 3.2 stderr в файл
~~~bash
ls /no_such_dir 2> err.txt
cat err.txt
~~~

### 3.3 stdout и stderr вместе (ВАЖНО: порядок)
Чтобы оба потока ушли в один файл:
~~~bash
ls /tmp /no_such_dir > all.txt 2>&1
~~~

Почему порядок важен:
- Bash настраивает редиректы слева направо.
- Поэтому `> all.txt 2>&1` и `2>&1 > all.txt` — разные команды.
Официальный пример про порядок редиректов есть в Bash Reference Manual. [web:25]

Также в bash есть сокращение:
~~~bash
ls /tmp /no_such_dir &> all.txt
~~~
Замечание: `&>` — bash-специфично; в `/bin/sh` может не работать.

---

## 4) Пайпы (|)
Пайп соединяет stdout одной команды со stdin другой.

Пример (посмотреть последние ssh-события в auth.log):
~~~bash
sudo tail -n 200 /var/log/auth.log | grep -i ssh | tail -n 20
~~~

---

## 5) QA-кейс: собрать диагностический “пакет” в два файла
Задача: собрать вывод диагностики в `out.log`, а ошибки — в `err.log`, чтобы прикрепить к тикету/CI.

~~~bash
mkdir -p ~/qa-diag
cd ~/qa-diag

( echo "== DATE =="; date; \
  echo "== UPTIME =="; uptime; \
  echo "== DISK =="; df -h ) > out.log 2> err.log

ls -lah
echo "---- out.log ----"
sed -n '1,120p' out.log
echo "---- err.log ----"
sed -n '1,120p' err.log
~~~

---

## 6) Задания (с проверкой)
1) Раздели stdout и stderr команды ниже в разные файлы `out.txt` и `err.txt`:
~~~bash
ls /tmp /no_such_dir
~~~

2) Сделай так, чтобы обе части (stdout+stderr) попали в `all.txt` (одним файлом).
Подсказка: `> all.txt 2>&1`. Поясни словами, почему порядок важен. [web:25]

3) Собери файл `failed_ssh.log` — последние 50 строк из auth.log, где есть `Failed`:
~~~bash
sudo grep "Failed" /var/log/auth.log | tail -n 50 > failed_ssh.log
~~~

---

## 7) Мини-инцидент (как на работе)
Симптом: “в CI тест упал, а в логе только ‘Command failed’”.

Твоя задача:
- запусти команду так, чтобы stderr сохранился в отдельный файл артефакта,
- приложи `err.log` к тикету,
- по `err.log` сформулируй 1–2 гипотезы причины (доступы, отсутствует файл, сеть и т.д.).
